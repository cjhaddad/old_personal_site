"I'm Falsy!"

Falsiness and Truthiness is JavaScript and Ruby

March 22, 2015

The title of this blog post - "I'm Falsy!" -- can be sung to the tune of Kelis's "I'm Bossy!" I figured it was due time for another questionably awesome blog post title that references a fabulous song form the not-so-distant past (See my first post "Gettin' Gittu Wit It"). Kelis's "Bossy" begins with the lyrics: "You don't have to love me, you don't even have to like me. But you will respect me." Falsiness and truthiness in any language are kind of bossy, and so it seems pretty appropriate that this song popped into my head as I began to think of writing this post. And whether or not you love, like, or dislike the ways truthiness and falsiness work in JS or Ruby -- you got to respect them.

So before we even dive into the differenes between truthiness and falsiness in Ruby and JavaScript. Let's talk about what the heck truthiness and falsiness even mean. Every variable has a boolean value, meaning every variable can be considered either true or false, even when it is not explicitly a boolean true or false value. Javascript and Ruby have some overlap in what kinds of values are considered true or false. Yet, they also have some differences. This is super important to understand, because control flow and various loops and such depend on a condition being either true or false. If you are providing a condition something that is considered false, when you want it to be considered true, you're going to run into some snags. That's why it's important to remember how truthiness and falsiness work in the language you are using. Since Ruby and JavaScript are the languages we're primarily learning at DBC, I'm focusing on these for now.

I think the clearest way to remember what values are truthy and what values are falsy are to focus what is considered falsy. Javascript has more values that are considered falsy, than Ruby does. In Ruby a false boolean value of course evaluates to false, but so does nil. These are the only two values that evaluate to false in Ruby. 

        
          x = nil

          if x == true
            p "nil is true!"
          else
            p "nil is false!"
          end

          >> "nil is false!"

In JavaScript there are number of other variable values that evaluate to false. One of the reasons I see this occuring is because JavaScript has more values in Ruby that stand for something that relates to the idea of being unknown, if you will. Ruby has nil, whereas JavaScript has undefined, NaN and null -- all of which evaluate to false, and are therefore considered falsy. A JavaScript variable is undefined when it is does not have a value. When a variable is undefined, the typeof variable it is is also undefined. This is important to note when you consider what null means. When a variable is null, it means it is "nothing." However, the typeof variable it is evaluates to being an object (See W3School). The NaN value means not-a-number. All three of these values -- undefined, null, and NaN -- hold a falsy value and therefore evaluate to false in a boolean context. 

        
          var x = null;
          if (x){
            console.log( x + " is Truthy!");
          }
          else{
            console.log( x + " is Falsy!");
          };
          >> null is Falsy!


          var y = undefined;
          if (y){
            console.log( y + " is Truthy!");
          }
          else{
            console.log( y + " is Falsy!");
          };
          >> undefined is Falsy!


          var z = NaN;
          if (z){
            console.log( z + " is Truthy!");
          }
          else{
            console.log( z + " is Falsy!");
          };
          >> NaN is Falsy!

In addition to null, undefined, and NaN, 0 and "" (an empty string) also evaluate to false in JavaScript. 0 and "" evaluate to true in Ruby -- so the boolean context of these two values is particularly helpful to remember. 

        
           if (0){
            console.log(" 0 is Truthy!");
          }
          else{
            console.log( "0 is Falsy!");
          };
          >> 0 is Falsy!

          if (""){
            console.log("An empty string is Truthy!");
          }
          else{
            console.log( "An empty string is Falsy!");
          };
          >> An empty string is Falsy!
        
      
        
          x = ""
          if x
            p "An empty string is Truthy!"
          else
            p "An empty string is Falsy!"
          end

          >> An empty string is Truthy!


          if 0
            p "0 is Truthy!"
          else
            p "0 is Falsy!"
          end

          >> "0 is Truthy!"

The code snippets above show how an empty string "" and 0 evaluate differently in JavaScript and Ruby. Similar differences in evaluations between various, as I'll call them, ambiguous values exist in other programming languages as well. To philosopher in me, I think this variety in how "truth" and "falsehood" are defined speaks to the subjectivity of truth. It's fun to see how this plays out in truthiness and falsiness in various languages.
